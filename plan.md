# Go 学习计划：从高级前端到通用 Go 工程师

这是一条 **13 周（第 0–12 周）**、每周 8–12 小时的可执行路线。它利用你已有的 JavaScript、TypeScript、Node.js 工程经验，同时刻意训练 Go 独有的类型设计、显式错误、并发、I/O、系统与网络编程能力；内容不局限于 Web。

仓库中的配套材料位于 `daily/YYYY-MM-DD`。日期是每周一；每个目录都是一周的学习单元，包含说明、可运行示例、练习和测试。

从 [daily/README.md](daily/README.md) 进入可查看完整课程索引。建议始终从当周 README 的“开始前诊断/任务梯度”开始，而不是直接照抄实现。

## 1. 最终能力目标

完成后，你应该能够：

- 写出符合 Go 惯例的包、接口、错误处理和测试，而不是“用 Go 语法写 Node.js”；
- 根据所有权和数据流选择 goroutine、channel、mutex、atomic 与 context；
- 独立开发 CLI、文件处理器、网络程序、后台任务和数据处理工具；
- 使用测试、race detector、benchmark、pprof、结构化日志定位正确性与性能问题；
- 阅读标准库及常见 Go 项目，并交付一个可发布、可观察、可维护的中等规模项目。

## 2. 执行规则

### 时间与比例

每周投入 8–12 小时：

- 学（30%）：官方文档、标准库、小实验；
- 练（35%）：3–5 个聚焦单点的练习；
- 做（35%）：一个能运行、能测试、能解释的周交付物。

忙碌周保底 4 小时：完成 README 中标为“核心”的任务、通过测试、写一段复盘。不要靠下一周双倍补课。

### 每周固定节奏

| 时间 | 行动 | 必须留下的证据 |
| --- | --- | --- |
| 周一 | 阅读本周 README，运行现有测试，记录预测 | 一条“我以为 / 实际是” |
| 周二 | 学语言或标准库，做最小实验 | 可运行代码 |
| 周三 | 完成练习 1–2 | 测试和错误分支 |
| 周四 | 完成练习 3 或重构 | 表驱动测试 / race / benchmark |
| 周五–周六 | 完成周项目 | README 使用示例 |
| 周日 | 验收、复盘、列下周风险 | 150–300 字复盘 |

### 每周统一验收

在对应周目录执行：

```bash
gofmt -w .
go test ./...
go vet ./...
```

涉及并发的周额外执行：

```bash
go test -race ./...
```

性能周额外执行：

```bash
go test -bench=. -benchmem ./...
```

“看懂示例”不算完成。至少要做一次行为修改、补一个测试，并能解释选择背后的取舍。

## 4. 分阶段学习重点

### 第 0 周：环境与 Node → Go 迁移

学习 `go mod`、`go run`、`go test`、`go vet`、`gofmt`、`go build`。用文件读取、HTTP 客户端、JSON 解析建立第一印象。

迁移重点：

- rejected Promise / exception → 显式返回并包装 `error`；
- 默认异步 I/O → 先写清晰同步代码，需要并发时显式引入；
- npm 包优先 → 先搜索标准库；
- class 层次 → 数据结构、小接口与组合。

### 第 1–2 周：语言基础与 API 设计

学习基本类型、零值、array、slice、map、string/rune、函数、多返回值、defer、struct、方法、interface、包和可见性。

必须理解：

- slice 是描述符，不是 JS Array；共享底层数组会产生别名问题；
- map 读取有零值与 `ok`，遍历顺序不稳定；
- interface 通常由使用方定义，并保持小而专；
- 用 `errors.Is/As` 保留错误语义，不靠字符串判断；
- 先写具体实现，有真实的第二种类型后再抽象。

### 第 3 周：测试、调试与质量

掌握表驱动测试、子测试、测试 helper、依赖注入、benchmark、覆盖率和 race detector。mock 只用于真正的外部边界，优先传入函数、小接口、`io.Reader/io.Writer`。

验收不仅看覆盖率；至少覆盖正常路径、边界、无效输入和底层依赖失败。

### 第 4–5 周：并发

学习 goroutine、channel、select、WaitGroup、Mutex/RWMutex、atomic、context、worker pool、fan-out/fan-in、限流与背压。

判断规则：

- channel 适合任务流、结果流、所有权转移；
- mutex 适合保护一组共享状态；
- 启动 goroutine 前先回答“谁等待它、谁让它退出”；
- 不在结构体中保存请求级 context；
- 并发正确性的最低证据是 `go test -race ./...` 和取消测试。

### 第 6–7 周：CLI、系统与网络

把 `io.Reader/io.Writer` 当作核心架构边界。CLI 要支持 stdin/stdout、清晰的 stderr 与退出码；网络程序要明确消息边界、大小限制、deadline 和关闭语义。

目标不是套框架，而是理解字节如何流动、错误在哪里发生、资源由谁关闭。

### 第 8–9 周：数据、任务、性能与观测

学习存储边界、事务、原子写、幂等、重试、任务状态机，以及 `database/sql` 的连接池和 context。样例用标准库文件仓库保证离线可运行；完成核心任务后，可用 SQLite/PostgreSQL 实现同一接口进行扩展。

性能坚持四步：建立基线 → 定位 → 单点修改 → 再测。结构化日志记录事件，不拼接难以检索的长字符串。

### 第 10–12 周：综合项目

默认项目是“本地优先文件同步工具”，覆盖文件系统、哈希、并发、CLI、取消、原子写、性能和可观察性。三个里程碑逐步增加能力：

1. 扫描目录并生成稳定 manifest；
2. 比较 manifest、生成计划、执行复制与校验；
3. 增加 CLI、dry-run、结构化日志、集成测试、CI 和发布说明。

可替换题目：任务调度器、日志采集管道、局域网文件传输、简易消息队列、代码仓库分析器。若替换，仍必须覆盖同等工程能力。

## 5. 综合项目完成定义

- README 含问题范围、非目标、安装、示例和限制；
- 包边界清楚，入口与核心逻辑分离；
- 关键逻辑有单元测试，主流程有集成测试；
- `go test -race ./...`、`go vet ./...` 通过；
- 长任务支持 context 取消和优雅退出；
- 文件更新采用临时文件 + rename，失败不破坏旧数据；
- 有结构化日志或最小指标；
- 有一次 benchmark / pprof 前后对比；
- CI 自动检查格式、测试、race 和 vet；
- 能构建至少一个目标平台的二进制并写发布说明。

## 6. 评分与复盘

每周 10 分，低于 7 分不急着进入下一周：

| 维度 | 分值 | 判定 |
| --- | ---: | --- |
| 正确性 | 3 | 测试覆盖正常、边界、错误路径 |
| Go 惯例 | 2 | 小接口、清晰 error、资源及时关闭 |
| 工程性 | 2 | 可运行命令、README、格式与 vet |
| 解释能力 | 2 | 能说明与 Node/TS 的差异和取舍 |
| 复盘 | 1 | 写下问题、证据和下一步 |

每周回答：

1. 本周最容易按 JS/TS 思路写错的地方是什么？
2. 哪个标准库抽象最值得复用？
3. 哪段代码的错误、资源或并发边界不清楚？
4. 哪个结论有测试、race 或 benchmark 证据？
5. 下周最小可交付结果是什么？

逐周章节、固定教材版本、阅读程度和完成证据见 [daily/reading-guide.md](daily/reading-guide.md)；不要只按下面的总书单从头通读。

3. [Effective Go](https://go.dev/doc/effective_go)
4. [Go Blog](https://go.dev/blog/)
5. [Go 标准库文档](https://pkg.go.dev/std)
6. 《The Go Programming Language》
7. 《Concurrency in Go》
8. 标准库源码：`io`、`context`、`sync`、`net`、`net/http`、`encoding/json`、`testing`

前端背景通常能很快掌握语法。真正决定 Go 水平的是：能否画清数据所有权、错误边界、资源生命周期与并发退出路径。因此，至少一半实践时间应投入这些部分，而不是只刷语法题或只搭 Web API。
